# Shared Library Versioning in the MFE Platform

> **TL;DR** — Change two lines in `federation-config/index.js` to switch every MFE
> between isolated versions (`singleton: false`) and a forced single version
> (`singleton: true`). The live proof is the **Instance ID** shown in each MFE panel.

## 1. Architecture Overview

```
┌─────────────────────────────────────────────────────────────────┐
│  @shared/federation-config  (separate repo, private registry)   │
│                                                                 │
│  sharedLibraries: {                                             │
│    '@shared/component-library': { singleton: false, ... },      │
│    '@shared/state-lib':         { singleton: false, ... },      │
│  }                                                              │
└────────────────────────┬────────────────────────────────────────┘
                         │  npm install @shared/federation-config
              ┌──────────┼──────────┐
              ▼          ▼          ▼
         shell-app  mfe-dashboard  mfe-settings
         (port 4200) (port 4201)   (port 4202)
              │          │          │
              └──────────┴──────────┘
                         │
              Each federation.config.js spreads
              ...sharedLibraries into its shared: {}
```

Every MFE's `federation.config.js` does:

```js
const { sharedLibraries } = require('@shared/federation-config');

module.exports = withNativeFederation({
  shared: {
    ...shareAll({ singleton: true, strictVersion: true, requiredVersion: 'auto' }),
    ...sharedLibraries,   // ← central overrides win (applied last)
  },
});
```

**One change in `federation-config/index.js` → all MFEs pick it up on next build.**

## 2. The Two Modes Explained

### `singleton: false` — Independent Versions (default)

```
shell-app          mfe-dashboard       mfe-settings
    │                   │                   │
    │  @shared/         │  @shared/         │  @shared/
    │  state-lib        │  state-lib        │  state-lib
    │  v1.0.0           │  v1.0.0           │  v1.0.0
    │  [Instance A]     │  [Instance B]     │  [Instance C]
    │                   │                   │
    ▼                   ▼                   ▼
 AppStateService    AppStateService    AppStateService
 notifications: 0  notifications: 0  notifications: 0
```

- Native Federation creates a **separate module scope** per MFE.
- `new AppStateService()` runs independently in each scope.
- Each MFE gets a **unique `instanceId`** (generated by `Math.random()` at construction).
- State changes in one MFE are **not visible** in another.
- Each MFE can install a **different version** of the library in its own `package.json`.

### `singleton: true` — Forced Single Version

```
shell-app          mfe-dashboard       mfe-settings
    │                   │                   │
    └───────────────────┴───────────────────┘
                        │
               @shared/state-lib
               v1.0.0  (loaded ONCE)
               [Instance A]
                        │
              ┌─────────┴─────────┐
              ▼                   ▼
         AppStateService     AppStateService
         (same reference)    (same reference)
         notifications: 3    notifications: 3  ← shared!
```

- Native Federation ensures **only one copy** of the library is loaded.
- The first MFE to load wins; subsequent MFEs reuse that instance.
- All MFEs show the **same `instanceId`**.
- State changes in one MFE are **immediately visible** in all others.
- All MFEs **must declare the same version** in their `package.json`.
  With `strictVersion: true`, a version mismatch throws at runtime.

## 3. Live Demonstration — Instance ID

`AppStateService` exposes two diagnostic properties added specifically for this demo:

```typescript
// shared-state-lib/src/lib/app-state/app-state.service.ts

readonly instanceId = Math.random().toString(36).slice(2, 8).toUpperCase();
// Generated ONCE when the service is constructed.
// Same ID across MFEs = singleton. Different IDs = non-singleton.

readonly libVersion = STATE_LIB_VERSION;  // '1.0.0'
```

### What to look for

| Where | What you see | Meaning |
|---|---|---|
| Shell navbar | `Shell:XXXXXX` badge | Shell's own `AppStateService` instance |
| Dashboard MFE panel | Instance ID box | Dashboard's instance |
| Settings MFE panel | Instance ID box | Settings' instance |

**`singleton: false` (current default):**

```
Shell navbar:    Shell:A1B2C3
Dashboard MFE:        D4E5F6   ← different from shell
Settings MFE:         G7H8I9   ← different from both
```

Increment notifications in Dashboard → Dashboard shows 1, Settings shows 0.

**`singleton: true` (after switching):**

```
Shell navbar:    Shell:A1B2C3
Dashboard MFE:        A1B2C3   ← SAME as shell
Settings MFE:         A1B2C3   ← SAME as shell
```

Increment notifications in Dashboard → Dashboard shows 1, Settings shows 1, Shell badge shows 1.

## 4. How to Switch Modes

### Step 1 — Edit `federation-config/index.js`

```js
sharedLibraries: {

  '@shared/component-library': {
    singleton: true,        // ← was false
    strictVersion: true,    // ← was false
    requiredVersion: 'auto',
  },

  '@shared/state-lib': {
    singleton: true,        // ← was false
    strictVersion: true,    // ← was false
    requiredVersion: 'auto',
  },
},
```

### Step 2 — Rebuild the state library

```bash
cd shared-state-lib
npm run build
```

### Step 3 — Reinstall in all MFEs (picks up new dist)

```bash
cd ../shell-app    && npm install
cd ../mfe-dashboard && npm install
cd ../mfe-settings  && npm install
```

### Step 4 — Rebuild all MFEs

```bash
cd ../shell-app    && npm run build
cd ../mfe-dashboard && npm run build
cd ../mfe-settings  && npm run build
```

### Step 5 — Verify

Open the shell app. Navigate to Dashboard and Settings.
Both panels should show the **same Instance ID**.
Increment notifications in one — the count updates in both and in the shell navbar.

## 5. Per-MFE Version Overrides

A single MFE can override the central config for its own use.
The local entry wins because it is spread **after** `...sharedLibraries`:

```js
// mfe-dashboard/federation.config.js

const { sharedLibraries } = require('@shared/federation-config');

module.exports = withNativeFederation({
  shared: {
    ...shareAll({ singleton: true, strictVersion: true, requiredVersion: 'auto' }),
    ...sharedLibraries,

    // ── Per-MFE override ─────────────────────────────────────────────────
    // This MFE stays on v1.x while the platform moves to v2.x.
    // Remove this block once the MFE is upgraded.
    '@shared/component-library': {
      singleton: false,
      strictVersion: false,
      requiredVersion: '^1.0.0',   // pin to v1.x
    },
  },
});
```

This is the **gradual migration path**: the platform can move to `singleton: true` globally
while one MFE temporarily opts out until it is ready to upgrade.

## 6. Decision Guide — Which Mode to Use

| Library | Recommended mode | Reason |
|---|---|---|
| `@angular/core` | `singleton: true, strictVersion: true` | Angular **requires** a single instance. Multiple copies break change detection. |
| `@shared/state-lib` | `singleton: true, strictVersion: true` | State must be shared across MFEs for a coherent user experience. |
| `@shared/component-library` | `singleton: false` (start) → `true` (mature) | UI components are safe to run in isolation. Move to singleton once all MFEs are on the same version. |
| Third-party UI libs (Bootstrap, etc.) | `singleton: false` | Each MFE can use its own version without side effects. |
| RxJS | `singleton: true` | Shared observables and subjects require a single instance. |

### Rule of thumb

> **If the library holds shared state or registers global singletons → `singleton: true`.**
> **If the library is purely functional/presentational → `singleton: false` is safe.**

## 7. Troubleshooting

### "Singleton mismatch" error at runtime

```
Error: Unsatisfied version constraint for @shared/state-lib.
Required: ^1.0.0, got: 2.0.0
```

**Cause:** `strictVersion: true` is set and one MFE has a different version in its `package.json`.

**Fix:** Align all MFE `package.json` files to the same version, or temporarily set
`strictVersion: false` while migrating.

---

### All MFEs show the same Instance ID but state is not shared

**Cause:** The library is `singleton: true` but the MFEs are running from separate dev servers
that each loaded their own copy before federation negotiation completed.

**Fix:** Always test singleton behaviour with production builds (`npm run build`), not `ng serve`.
Dev server HMR can interfere with federation module negotiation.

---

### Instance IDs are different even with `singleton: true`

**Cause:** One or more MFEs still have the old dist in `node_modules`.

**Fix:** Run `npm install` in every MFE after rebuilding `shared-state-lib`, then rebuild all MFEs.

---

### TypeScript errors: `Property 'instanceId' does not exist on type 'AppStateService'`

**Cause:** The MFE's `node_modules/@shared/state-lib` has the old dist (before `instanceId` was added).

**Fix:**
```bash
cd shared-state-lib && npm run build
cd ../mfe-dashboard && npm install
cd ../mfe-settings  && npm install
cd ../shell-app     && npm install
```
